정리해 주면 네가 앞으로 포트폴리오 만들 때 갈등할 부분 대부분이 사라질 거다.

지금까지 나온 내용 기준으로 “너에게 가장 잘 맞는 전략 세트”를 실행 순서대로 묶어서 정리한다.

---

✅ 전체 전략 요약 — Obsidian → Astro → GitHub Pages

1. 콘텐츠 (글/이미지) 중심 관리: Obsidian 기반 폴더 구조

프로젝트 하나를 문서 단위로 관리한다.

projects/

  project-a/

    [index.md](http://index.md/)

    main.png

    ui-1.png

  project-b/

    [index.md](http://index.md/)

    main.png

Markdown 과 이미지가 같은 폴더에 묶여 있음

Obsidian 에서 드래그로 이미지 추가 → 상대 경로 자동 삽입 → Astro 에서도 그대로 사용 가능

관리 포인트가 “문서 + 관련 이미지 묶음”으로 단순해진다

의도

→ 이미지/문서 링크 깨지는 문제 제거

→ 프로젝트 단위로 보관/삭제/수정이 깔끔해짐

---

2. Astro 를 페이지 프레임워크로 사용

Astro 는 정적 페이지 생성이 주 역할이고, Markdown 을 바로 페이지로 변환할 수 있다.

Markdown → Astro content collection 으로 불러오기

프로젝트 목록 페이지에서 자동으로 나열

React 는 필요한 부분만 컴포넌트로 사용

의도

→ Markdown 중심 흐름 유지

→ Astro 는 “그걸 보기 좋게 렌더링하는 레이어” 역할에 집중

→ 프론트엔드 실력 어필하려면 React 일부 섞는 방식 가능

---

3. React 는 꼭 필요한 부분만 최소 삽입

Astro 페이지 안에 특정 UI 요소만 React 로 작성한다.

예:

프로젝트 카드 UI

토글/탭

검색/필터

복잡한 앱 형태가 아니면 React 전체 도입은 낭비고, “부분적 사용”이 효율적이다.

의도

→ Astro 의 간결성과 React 의 편의성을 동시에 취함

→ 전체를 React 로 만들 때 발생하는 번거로운 빌드/라우팅 관리 피함

---

4. 이미지 관리 방식 — Markdown 옆에 두기 고정

이미지 관리 전략의 핵심:

이미지 링크: ./main.png 형태의 상대 경로

Obsidian 에서도 자연스럽게 처리됨

Astro 빌드 시에도 이미지 자동 포함 가능 (설정 매우 간단)

의도

→ 이미지 경로 관리 스트레스 제거

→ Issue/폴더 분기 고민 불필요

→ “문서 하나 = 프로젝트 전체” 구조 유지

---

5. 배포는 GitHub Pages + GitHub Actions 자동화

전형적인 흐름:

1. Astro 로 빌드
2. /dist 생성
3. GitHub Actions 가 자동으로 Pages 브랜치로 배포

의도

→ 로컬에서 git push 만 하면 자동 배포

→ 별도 서버/도메인/인프라 신경 쓸 필요 없음

---

6. 유지 관리 방식

3 개월 주기로 아래만 체크한다:

1. 신규 프로젝트 Markdown 추가
2. 이미지 폴더에 스크린샷 넣기
3. 썸네일만 하나 지정하면 리스트 페이지에 자동 반영
4. 오래된 프로젝트 폴더는 통째 삭제

의도

→ “생산성 유지”를 우선

→ 파일 기반 구조라 Obsidian 에서 그대로 편집 → Astro 에서 그대로 빌드

---

🎯 전체 전략을 한 문장으로 요약하면:

> Obsidian 에서 Markdown + 이미지 한 폴더로 관리 → Astro 가 그걸 그대로 읽어와 페이지 생성 → React 는 필요한 부분만 → GitHub Pages 로 자동 배포.

너의 성향 (Obsidian 많이 씀, Markdown 기반 작업 선호, 이미지 관리 귀찮음 극혐, 실용 위주) 과 완벽하게 맞아떨어지는 조합이다.
