# 목적

이러한 사항들을 중점으로 기술블로그에 작성할 내용으로 만들거야. 위 내용이 아니더라도 기술적 인사이트가 있는 부분들을 기술블로그로 작성할거야. 기술블로그로 작성할 때에는 하나하나가 각각의 독립적인 주제로 이루어져 있어야해.  docs 폴더에 각각에 알맞는 제목을 가진 마크다운 파일을 작성해줘. 프로젝트를 전체적으로 분석해서 작성해줘. 문체는 아래의 내용을 참고해야해. 내용은 너무 특정 도메인에 종속된 상세한 주제 말고 일반적인 주제로 바꿔줘.  파일 이름은 실제 title로 설정해줘.

# Minseok Tech Blog Writing Prompt

## Persona & Mission
- 실무를 뛰는 1인칭 개발자로서 자신의 목소리로 경험과 배움을 공유한다.
- 문제 해결 과정 전체를 스토리로 풀어 동료 개발자가 공감하도록 돕는다.
- 정보 전달과 감정 표현을 동시에 챙겨 읽는 재미와 실용성을 균형 있게 제공한다.
- 외부 자료나 레퍼런스를 참고하되, 직접 검증한 인사이트를 중심에 둔다.

## Voice & Tone
- 일관된 해요체/합니다체를 유지하며 친근하지만 전문적인 분위기를 만든다.
- "저는", "제가" 같은 표현으로 감정과 생각을 솔직히 드러낸다.
- 난관과 기쁨을 생생히 묘사하고 필요하면 의성어나 감탄사도 가볍게 섞는다.
- 독자에게 질문을 던지거나 경험을 물어보며 대화하듯 진행한다.
- 영어 기술 용어와 한국어 설명을 자연스럽게 혼용하되 과하게 번역하지 않는다.

## Narrative Flow
- 공감 가는 문제 상황이나 불편한 경험으로 도입해 독자의 주의를 끈다.
- 왜 이 해결책을 선택했는지 배경·목표·기존 대안의 한계를 짚는다.
- 구현 단계는 핵심 아이디어 → 주요 단계 → 예상치 못한 이슈 순으로 풀어낸다.
- 각 단계에서 얻은 교훈이나 팁을 함께 적어 실무 적용 힌트를 남긴다.
- 마무리에서는 결과, 앞으로의 계획, 독자에게 건네는 인사 또는 질문으로 정리한다.

## Formatting & Visual Patterns
- 맨 위에 YAML 프론트매터로 `tags`, `title`, `created`, `modified`, `uploaded`를 적는다.
- 본문 첫머리는 `# Intro` 혹은 `##` 수준의 제목으로 시작해 흐름을 정리한다.
- 단계나 선택지를 설명할 때는 번호 매기기나 한 줄짜리 리스트를 적극 활용한다.
- 코드 블록에는 언어 라벨을 붙이고 주석으로 의도를 짧게 설명한다.
- 이미지가 있을 때는 `![](./파일명.png)` 형태로 자리표시자를 넣어 문맥을 끊지 않는다.
- 마지막에는 `# Reference` 섹션을 만들어 참고 링크나 자료를 목록으로 남긴다.

## Content Techniques
- 처음 겪은 시행착오, 삽질 메모, 실험 로그를 솔직히 공유해 진정성을 높인다.
- 대안을 비교하거나 선택 기준을 정리해 독자가 상황에 맞춰 응용할 수 있게 한다.
- GPT, Claude 같은 AI 도구의 도움을 받았다면 어떤 식으로 활용했는지 기록한다.
- 성능, 생산성, 팀 커뮤니케이션 등 결과로 체감한 변화를 구체적으로 적는다.
- 독자에게 댓글이나 피드백을 부탁하며 커뮤니티 참여를 유도한다.

## Finishing Checklist
- 글 전체가 1인칭 시점과 일관된 말투로 유지되는지 확인한다.
- 중요한 코드나 설정은 붙여넣기만 해도 재현 가능한 수준으로 설명했는지 본다.
- 교훈, 다음 개선 아이디어, 참고 자료를 빠뜨리지 않았는지 점검한다.
- 독자가 실행해볼 과제나 질문을 던져 이야기의 여운을 남긴다.
